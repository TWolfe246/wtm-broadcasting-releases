Building your own overlay.

If you want a custom timing tower or a custom overlay rendered in OBS, this is everything you need to know. The data pipe is small and the wiring is short. You should be running by the end of this page.

How it connects

Every overlay (built-in or custom) talks to one local WebSocket. The launcher boots the iRacing server, which pushes the full broadcast state to every connected client at 20Hz. 

ws://localhost:9211

There's no auth, no handshake, no event filtering. You connect, you get a JSON message every 50ms with the entire current state. Parse it, render the parts you care about.

Your file lives at %APPDATA%\wtm-launcher-electron\extensions\overlays\{' '} (for overlays) or extensions\towers\ (for tower variants). Drop an HTML file in, hit Refresh in the Customize tab, and the launcher gives you a URL to paste into an OBS browser source.

Hello world

The smallest working overlay. Drop this in as hello.html:

<!DOCTYPE html>
<html>
<body style="background: black; color: lime; font: 28px monospace; padding: 24px;">
  <div id="out">Connecting...</div>
  <script>
    const ws = new WebSocket('ws://localhost:9211');
    ws.onmessage = (event) => {
      const data = JSON.parse(event.data);
      const leader = data.standings?.[0]?.driver || 'no leader';
      const session = data.session?.type || 'no session';
      document.getElementById('out').textContent = session + ' · ' + leader;
    };
    ws.onclose = () => document.getElementById('out').textContent = 'disconnected';
  </script>
</body>
</html>

Add this as an OBS browser source pointed at the URL the launcher gives you. You'll see the session type and the current leader update in real time when iRacing is running.

The payload

Every message from the WebSocket is one JSON object with this shape. Everything under ? is optional — fields are populated as the session has data for them.

Top level

FieldTypeWhat it is
connectedbooleanTrue when the server has live iRacing telemetry.
trackobject · nullCurrent track info (just name right now).
sessionobject · nullCurrent session state. See session below.
total_carsnumberHow many cars are in this session.
standingsarrayEvery car, position-sorted. See standings row below.
tower_standingsarraySame shape as standings but post-race ordered by official results when available.
starting_gridarrayCars sorted by qualifying. Falls back to driver list pre-qual.
starting_grid_statusobject{ ready, source } — is the grid populated and from where.
configobjectSnapshot of the operator's launcher config. Useful for theme/style settings.
fastest_lapobject · nullCurrent overall fastest lap of the race.
fastest_lap_eventobject · nullFires when someone sets a new fastest lap. Watch seq to react.
cam_group_numnumberCurrent iRacing camera group.
cam_groupsarrayAvailable camera groups: { group_num, name }.
live_stageobjectStage progress: { completed: [{ stage }] }.
eventsobject · nullRolling event log (overtakes, off-tracks, crashes, flags). See below.
brandingobject{ series_name, series_logo, iracing_paint_path }.

events

The event log is a rolling array of things that happened. Each event has a monotonically increasing seq. To react to new events, track the highest seq you've seen and process anything newer than that.

FieldTypeWhat it is
seqnumberTop-level: highest event seq so far.
logarrayThe events themselves. Each has { seq, type, session_time, session_num, lap, car_idx, car_num, driver, detail }.

Event type is one of: overtake,{' '} off_track, crash, flag,{' '} pit_in, pit_out.

Common patterns

Render the standings list

ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  const rows = (data.standings || []).map((row) =>
    `<li>P${row.pos} ${row.driver} ${row.gap_fmt || ''}</li>`
  ).join('');
  document.getElementById('list').innerHTML = `<ol>${rows}</ol>`;
};

Find the focused driver

const focused = (data.standings || []).find((row) => row.is_focused);
if (focused) {
  showLowerThird(focused.driver, focused.car_num);
}

React only to new race events (not every tick)

let lastSeq = 0;
ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  const events = data.events?.log || [];
  for (const e of events) {
    if (e.seq > lastSeq) {
      // new event — fire your handler
      handleEvent(e);
      lastSeq = e.seq;
    }
  }
};

Detect a session change (reset your state)

let lastSubsession = null;
ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  const subsession = data.session?.subsession_id;
  if (subsession && subsession !== lastSubsession) {
    resetOverlay();
    lastSubsession = subsession;
  }
};

Heads up

Schema can change between releases.

Right now the WebSocket schema isn't versioned. When new fields get added you'll see them; when old ones change shape your overlay might break. Test your overlays against new releases before race night.

Don't compute race truth in your overlay.

The launcher already computes positions, gaps, intervals, lap-down status, and fastest lap. If you find yourself recomputing any of these client-side, you'll drift from the built-in overlays. Use the values already in the payload.

Custom tower variants get the same payload.

A tower variant is just another HTML file that connects to the same WebSocket. If your variant is loaded via the Customize tab's "Activate" button, it gets the same data feed as the built-in tower — no special wiring needed.

The launcher serves your files at HTTP/9212.

Custom overlays live at http://localhost:9212/custom/overlays/your-file.html{' '} and custom towers at /custom/towers/your-file.html. The Customize tab gives you the exact URL — click Copy and paste into your OBS browser source.
